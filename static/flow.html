Data Flow — Step by Step
1Alpaca SIP→trade tick→trade_queue→broadcast WS→Browser
2Alpaca News→article→news_queue→broadcast WS→Browser
3Browser computes VWAP 1m vs 2m every 5s→Δ% exceeds threshold→Alert fires
4Alert fires→POST /log/alert→alerts_log.jsonl (disk, persists)
5User clicks ticker→GET /bars/{sym}→historical + live merged→Chart rendered
6User clicks ticker→GET /news/{sym}→7-day history + live cache→News panel
7User clicks 🐦 Fetch X Posts→GET /twitter/{sym}→xAI x_search (last 2h)
8xAI returns tweets→summarize()→cache→Twitter card injected in news panel
9Background thread→POST /auth/refresh (JWT via cookie)→access token cached
10Background thread→POST /news→article published in Pulszy (source: PULSZY)
Key Design Decisions
🔒 Per-client WS Lock
broadcast() and per-client ping share an asyncio.Lock to prevent concurrent write crashes (Windows ProactorEventLoop fix)
📋 Alerts Only Logging
Only alert events are written to disk — not every trade. Log tagged "new" or "escalation" (delta grew 1.5× + 2× threshold)
🐦 Twitter On-Demand
X fetch is manual (button) not automatic — avoids wasting xAI API calls. Cache is 30 min; force-refresh always clears it
⚡ Non-blocking Push
Pulszy push runs in a daemon thread — the /twitter response returns to the browser immediately without waiting for Pulszy
🔄 JWT Auto-refresh
Pulszy token cached in-memory with 30s expiry buffer. On expiry, re-runs /auth/refresh with the cookie from .env automatically
🕐 Latest Tweets Anchor
Current ET time injected into xAI prompt — Grok searches last 2h first, expands to 24h only if fewer than 3 posts found
